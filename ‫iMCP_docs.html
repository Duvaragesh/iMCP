Concept

How iMCP Works

iMCP sits between your AI assistant and your backend services. It dynamically discovers operations from service contracts, generates structured tool definitions, and brokers authenticated calls — completely transparent to both sides.

1

Register a Service

Provide a WSDL URL, OpenAPI spec URL, or upload an MCP JSON file. No changes needed on the backend system.

2

Spec Parsing & Tool Generation

iMCP parses the spec, extracts all operations, converts XSD/JSON Schema types, and generates strongly-typed MCP tool definitions — one tool per operation.

3

Tools Cached & Served

Generated tools are stored in the database and held in a TTLCache. Any connected AI client receives the full tool list instantly via tools/list.

4

AI Calls a Tool

When the AI issues a tools/call, iMCP validates inputs, injects authentication headers, calls the upstream service (SOAP or REST), and returns the normalized result.

5

Audit & Observability

Every call produces a structured audit event: correlation ID, actor, tool name, latency, and a sanitized (secrets-redacted) payload.

Architecture

Internal Component Architecture

iMCP is a Django application with three layers: views (HTTP/MCP protocol), services (business logic), and models (persistence). Every component has a single clear responsibility.

Workflow

Tool Call Request Flow

Every AI tool call passes through a consistent, audited pipeline. The same path is used by both the live MCP endpoint and the Admin Portal Test Console — guaranteed parity.

Spec Types

Three Ways to Register a Service

iMCP supports three spec formats. Choose based on what your backend exposes — no changes to the backend required.

WSDL / SOAP

Point iMCP at any WSDL URL or upload a .wsdl / .xml file from IBM AS400, SAP, Oracle, mainframe, or any enterprise platform.

  • ✓ Parses all services, ports, and operations
  • ✓ Converts XSD complex types → JSON Schema
  • ✓ Handles nested objects (policies, claims, etc.)
  • ✓ Operation allowlists / denylists
  • ✓ Works over HTTP and HTTPS
IBM AS400 SAP Mainframe Oracle

Example WSDL registration

# Register a SOAP service via Admin Portal or API
POST /imcp/admin/services
{
  "name": "PolicySearch",
  "spec_type": "wsdl",
  "spec_url": "http://ibmi-host/wsservices/PolicySearch?wsdl",
  "auth_type": "basic",
  "auth_config": {
    "username": "svcuser",
    "password": "••••••"
  },
  "category": "Policy"
}

OpenAPI 2.x / 3.x

Supply an OpenAPI spec as a URL or upload a .yaml / .json file. Works with any REST API — cloud, on-premise, or API gateway.

  • ✓ Full $ref resolution via prance
  • ✓ One tool per path + method combination
  • ✓ Maps path, query, and body params
  • ✓ Preserves enum, min/max, pattern constraints
  • ✓ Supports Apigee, AWS APIGW, Kong hosted specs

Example OpenAPI registration

POST /imcp/admin/services
{
  "name": "ClaimsAPI",
  "spec_type": "openapi",
  "spec_url": "https://api.internal/claims/openapi.yaml",
  "auth_type": "oauth2",
  "auth_config": {
    "token_url": "https://auth.example.com/token",
    "client_id": "imcp-client",
    "client_secret": "••••••",
    "scope": "read:claims"
  }
}

MCP JSON

A lightweight custom format for wrapping any HTTP endpoint directly — no full WSDL or OpenAPI spec required. Ideal for simple REST services or quick prototyping.

  • ✓ No spec file needed — describe tools directly
  • ✓ Supports GET / POST / PUT / DELETE
  • ✓ Query params and request body mapping
  • ✓ Default values for optional parameters
  • ✓ One-click template download from portal

MCP JSON format

{
  "name": "my-service",
  "tools": [{
    "name": "searchProduct",
    "description": "Search product catalog by SKU",
    "inputSchema": {
      "type": "object",
      "properties": {
        "sku": { "type": "string" }
      },
      "required": ["sku"]
    },
    "endpoint": {
      "method": "GET",
      "baseUrl": "https://api.example.com",
      "path": "/catalog/products",
      "queryParams": ["sku"]
    }
  }]
}
Authentication

Auth Proxy — Four Supported Types

iMCP handles all outbound authentication transparently. Credentials are Fernet-encrypted at rest and never exposed to AI clients or logs.

Bearer Token

Static token injected as Authorization: Bearer … on every upstream call.

{ "token": "your-token" }
Basic Auth

Username + password encoded as Authorization: Basic … on every request.

{ "username": "u", "password": "p" }
Custom Headers

Any arbitrary headers — X-API-Key, X-Tenant, or any proprietary header your backend requires.

{ "headers": { "X-API-Key": "k" } }
OAuth2 Client Credentials

Fetches a token from a token endpoint, caches it for ~58 minutes, and refreshes automatically. Fully transparent to the AI.

{
  "token_url": "https://…/token",
  "client_id": "id",
  "client_secret": "s",
  "scope": "read:api"
}

Security

Security Model

iMCP is built with security-by-default across every layer — credential storage, transport, logging, and operation governance.

🔐

Fernet Encryption at Rest

All service credentials (tokens, passwords, OAuth2 secrets) are AES-128-CBC encrypted using a key derived from Django's SECRET_KEY. Decrypted only in memory at call time.

🧹

PII Redaction in Logs

Configurable redaction patterns scrub Authorization headers, token, password, client_secret, and any custom PII fields from all audit records.

🎛️

Operation Governance

Per-service allowlists and denylists enforce which operations are exposed. Operations outside the allowlist are invisible at tools/list — undiscoverable and uncallable.

🗝️

Hashed API Keys

API keys are stored as hashed values only. They are never returned by any GET endpoint. Revocation is instant via the Token Manager.

📋

Structured Audit Trail

Every tool call, service change, and auth event is logged with: correlation_id, actor, action, service, tool, status, latency, and a redacted payload.

🌐

TLS + Rate Limiting

All upstream calls use httpx with TLS. Portal traffic should sit behind an Nginx TLS proxy. In-memory rate limiting (configurable per-minute limit).

Quick Start

Up in 5 Minutes

# 1. Clone and install
git clone https://github.com/your-org/imcp
cd iMCP
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# 2. Migrate database
python manage.py migrate

# 3. Create superuser (for admin portal)
python manage.py createsuperuser

# 4. Start server
python manage.py runserver

Then open http://localhost:8000/imcp/portal/

1

Add a Service

Go to Services → Add Service. Provide a WSDL URL, OpenAPI spec, or upload MCP JSON.

2

Verify Tools

Go to Tools. Inspect generated tool schemas and input definitions.

3

Test It

Open Test Console, select a tool, and run it against the real upstream.

4

Connect Your AI

Register the MCP endpoint in Claude Code or VS Code and start calling tools conversationally.
